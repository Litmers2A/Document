# Change Analysis: Episode/Proposal 관리 Story-first 레이아웃 전환

> 생성일시: 2026-01-13 14:30
> 상태: applied

## 원문 (Raw Feedback)

```markdown
# Episode / Proposal Management Layout Prompt

**Context**

* 하나의 Story 안에 다수의 Episode가 포함되는 구조
* Story 개수: 10~100+
* Story당 Episode 개수: 100+
* Episode당 Proposal 개수: 100+
* 대량 데이터 환경에서도 스크롤 폭증 없이 관리 가능해야 함

---

## 1️⃣ Episode 관리 페이지 (Creator Dashboard – Story-first)

Redesign the Episode Management Dashboard to be Story-first.

### Required Information Hierarchy

* **Story → Episode → Proposal**의 명확한 계층 구조를 유지할 것
* Dashboard 초기 진입 시 **Episode 리스트가 아닌 Story 리스트가 먼저 노출**

### Layout Structure

[ Story List ]  →  (Story 선택)
                     ↓
              [ Episode List for selected Story ]

### Story List (Primary View)

* 크리에이터용 **Story List Component** 사용
* 대량 Story(10~100+)를 고려한 구조

**Story List Item must support:**

1. **Navigate**: Story 클릭 시 → 해당 Story의 Episode 리스트 표시
2. **Edit**: Story 메타데이터 편집
3. **Preview**: Story 독자 관점 미리보기

**UI Constraints**

* Vertical list or table-based layout
* Search / filter / pagination or infinite scroll 지원
* 리스트 스크롤과 페이지 전체 스크롤은 분리

---

## 2️⃣ Episode List (Story Context 내부)

Display Episodes only after a Story is selected.

### Episode List Requirements

* 선택된 **Story에 속한 Episode만 표시**
* Episode 개수 100+를 전제로 한 성능 최적화

**we can use exisiting component**

---

## 3️⃣ Proposal 관리 페이지 (Episode-based Context)

Design Proposal Management as an Episode-scoped view.

### Required Layout

Left (or Top):
- Episode Selector (within the same Story)
  - Shows only OPEN episodes
  - Vertical list (desktop) / dropdown (mobile)
  - Searchable when Episode count is large

Main:
- Proposal List for the selected Episode only

### Proposal List Rules

* **한 번에 하나의 Episode에 대한 Proposal만 렌더링**
* Sorting / filtering / pagination or virtualized list 필수
* Proposal 수 100+에서도 성능 저하 없어야 함

---

## 4️⃣ Global Principles

* Never render Episodes or Proposals across multiple parent contexts
* Context switch는 항상 **Story → Episode → Proposal** 단방향
* Desktop-first creator tool UX를 기준으로 설계
* 확장성과 관리 효율을 시각적 화려함보다 우선
```
