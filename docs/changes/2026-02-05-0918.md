## 기술적 고려사항

### 1. 미들웨어 구현 방식
- **Next.js middleware.ts** 활용
- 온보딩 미완료 시 `/onboarding`으로 리다이렉트
- 예외 경로: `/auth/*`, `/onboarding`, `/api/*`

### 2. 인증 상태 동기화
- 로그인 직후 서버에서 `onboardingCompleted` 값 가져오기
- authStore에 저장하여 클라이언트에서 즉시 접근 가능
- 온보딩 완료 후 authStore 업데이트 + 페이지 리다이렉트

### 3. 휴대폰 인증 보안
- SMS 발송 횟수 제한 (예: 5분에 3회)
- OTP 유효 시간: 3분
- 인증 실패 횟수 제한 (예: 5회 실패 시 10분 대기)

### 4. 닉네임 중복 검사 최적화
- Debounce 적용 (500ms)
- 입력 중에는 검사하지 않음
- 입력 완료 후 자동 검사

### 5. 온보딩 이탈 방지
- 브라우저 뒤로가기 감지 및 차단
- 새로고침 시 온보딩 상태 유지
- 로그아웃 시에도 온보딩 상태 서버에 저장

## 리스크 및 대응 방안

### 리스크 1: 기존 사용자 마이그레이션
- **문제**: 기존 사용자들의 `onboardingCompleted` 필드가 `null` 또는 `false`
- **대응**: DB 마이그레이션 스크립트로 기존 사용자는 `onboardingCompleted = true` 설정

### 리스크 2: SMS 발송 실패
- **문제**: SMS 발송 실패 시 사용자 온보딩 진행 불가
- **대응**:
  - 재시도 버튼 제공
  - 실패 시 관리자 문의 안내
  - SMS 발송 로그 기록 (디버깅용)

### 리스크 3: OTP 인증 시간 초과
- **문제**: 사용자가 OTP 입력을 늦게 하는 경우
- **대응**:
  - 재전송 버튼 제공
  - 타이머 UI로 남은 시간 표시
  - 재전송 횟수 제한 (5분에 3회)

### 리스크 4: 온보딩 이탈
- **문제**: 사용자가 온보딩 중간에 브라우저를 닫는 경우
- **대응**:
  - 온보딩 진행 상태를 서버에 저장
  - 재로그인 시 진행 중이던 단계로 복귀
  - beforeunload 이벤트로 경고 메시지 표시

### 리스크 5: 접근 제어 미들웨어 버그
- **문제**: 온보딩 완료 체크 로직 오류로 서비스 접근 불가
- **대응**:
  - 철저한 테스트 (신규 사용자, 기존 사용자, 온보딩 중인 사용자)
  - 예외 경로 명확히 정의 (`/auth/*`, `/onboarding`, `/api/*`)
  - 에러 발생 시 로그아웃 유도
